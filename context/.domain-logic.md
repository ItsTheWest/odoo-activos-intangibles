# Domain Logic & Business Rules

This document outlines the core business logic, module integration strategies, and the lifecycle management of Intangible Assets within the system.

---

## 1. Module Architecture (Python Backend)

The Python backend is split into **single-responsibility files** following the OCA modular pattern:

```
gestion_activos_intangibles/
├── __init__.py                         # Loads: controllers, models, wizard
├── models/
│   ├── __init__.py                     # Import order: type → main → ir_attach → dashboard
│   ├── activo_intangible.py            # Core model: fields, compute, ORM, actions, cron
│   ├── activo_intangible_type.py       # Configuration catalogue: asset types & lifespan
│   ├── ir_attachment_inherit.py        # Extends ir.attachment: preview & delete confirm
│   └── dashboard.py                    # Read-only reporting model (OWL charts)
└── wizard/
    ├── __init__.py
    ├── activo_attachment_wizard.py     # Upload evidence files to an asset
    ├── activo_delete_wizard.py         # Confirm physical deletion of an attachment
    └── activo_near_expiry_wizard.py    # Confirm creation/activation of expiring asset
```

> **Rule**: Never add wizard classes to the `models/` directory. Wizards (`TransientModel`) live exclusively in `wizard/`.

## 2. Module Integrations (The Financial "Why")

Intangible assets do not exist in isolation. Their lifecycle is tied to legal and financial events managed by other Odoo modules.

### A. Invoicing (Vendor Bills): The Origin of Value
When a company purchases a software license, patent, or trademark, the transaction begins in the **Invoicing/Accounting** module.
- **The Concept**: A Vendor Bill records the legal debt (e.g., $5,000 owed to a provider).
- **The Relationship**: Unlike consumable expenses (like office supplies), these purchases represent an **Investment**.
- **Professional Workflow**: Validating a specific category of vendor bill should automatically trigger the creation of a **Draft Asset** in this module. This eliminates data entry redundancy and ensures financial alignment.

### B. Expenses: Operations and Maintenance
The "Day-to-Day" maintenance of an asset involves recurring costs such as renewal fees, legal filings, or maintenance contracts.
- **The Concept**: The **Expenses** module tracks these operational payments.
- **The Relationship**: These expenses must be linked back to the specific Intangible Asset record.
- **Business Value**: This allows for "Total Cost of Ownership" (TCO) reporting. The system can report: *"This trademark cost $1,000 to acquire, but has required $500 in maintenance over 2 years."*

## 3. Responsibility Matrix

| Module | Primary Function | Relation to Intangible Asset |
| :--- | :--- | :--- |
| **Invoicing** | Legal registration of cost. | Provides the **Initial Book Value** (Acquisition cost). |
| **Expenses** | Operational payment tracking. | Records **Maintenance & Renewal** costs over time. |
| **Assets (This Module)** | Lifecycle & Compliance control. | Manages owners, expiration dates, and proactive alerts. |

## 4. Core Business Rules

1. **Unique Identification**: Every asset must have a unique internal reference or serial number.
2. **State Transitions**:
   - **Activo**: Asset is legally valid and currently in use.
   - **Por Expirar**: Auto-triggered 60 days before the renewal date (by cron or on save).
   - **Expirado**: Renewal date has passed; requires urgent intervention.
   - **Inactivo**: Manually decommissioned or archived.
3. **Risk Level Computation** (`nivel_riesgo` — computed, stored):
   - **Alto**: Asset is `expirado`, OR is `por_expirar` with `valor_contable > 1100`.
   - **Medio**: Asset is `por_expirar` with `valor_contable <= 1100`.
   - **Bajo**: Asset is `activo` or `inactivo`.
4. **Near-Expiry Guard** (`ActivoNearExpiryWizard`): Any create or activate action that would place an asset in `por_expirar` state must be explicitly confirmed by the user via the wizard.
5. **Data Integrity**: Changes to `valor_contable` (Book Value) should be derived from financial documents (`account.move`) to ensure auditability. Direct edits are allowed but tracked via `mail.thread`.
